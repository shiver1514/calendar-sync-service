# 软件介绍

## 1. 产品定位

`calendar-sync` 是一个面向个人工作流的日历同步服务，目标是把“钉钉工作日程 + iCloud 个人日程”收敛到 iCloud 统一视图中，便于手机和后续 Agent 统一读取。

当前设计是严格单向：

`钉钉 CalDAV -> iCloud CalDAV`

即工作日程只从钉钉流向 iCloud，不反向写回钉钉。

## 2. 解决的问题

1. iOS 订阅钉钉日历只能“订阅显示”，不能和 iCloud 日历形成统一数据层。
2. 后续 Agent 若直接对接多个日历接口，复杂度高、容错差。
3. 需要一个可在 Linux 服务器长期运行、可观测、可运维的同步服务。

## 3. 功能概览

1. 同步能力：
   - 新增同步
   - 更新同步
   - 删除同步（保守策略）
   - 幂等同步（重复执行不产生脏写）
2. CLI 能力：
   - 同步控制：`sync once`
   - 状态读取：`status`
   - 日志读取：`logs`
   - iCloud 事件 CRUD：`icloud list/create/update/delete`
3. 服务能力：
   - 定时同步
   - 手动触发同步 API
   - 状态/日志/配置 API
4. UI 能力：
   - 状态卡片
   - 最近同步统计
   - 日志查看
   - 脱敏配置查看
5. Agent 技能能力：
   - 提供 `../skills/icloud-calendar-ops`，可直接执行 iCloud 事件 CRUD 与运行分析。

## 4. 技术架构

### 4.1 组件划分

1. CalDAV 通讯层：`src/lib/caldav.js`
2. 同步引擎：`src/sync/sync-once.js`
3. 状态存储：`data/sync-state.json`（映射与删除确认计数）
4. CLI 入口：`src/cli.js`
5. 服务入口：`src/server.js`
6. 日志模块：`src/lib/logger.js`
7. 前端页面：`src/ui/*`

### 4.2 数据流

1. 从钉钉按时间窗读取事件。
2. 根据 UID 与映射状态判断“新建/更新/跳过”。
3. 写入 iCloud 目标日历。
4. 对源端消失事件执行删除候选评估：
   - 连续 missing 计数
   - 删除比例保护
   - 源端二次探测
5. 持久化同步状态与日志。

## 5. 安全与防误删策略

1. `SYNC_DRY_RUN`：只演练不写入。
2. `SYNC_DELETE_CONFIRM_RUNS`：需连续 N 轮 missing 才可删除。
3. `SYNC_DELETE_MAX_RATIO`：单轮删除比例过高时整轮跳过。
4. 源端探测支持钉钉 `STATUS:CANCELLED` 语义，避免误判。

## 6. 适配 Agent 的原因

1. CLI 接口稳定，易封装为 Agent skill/action。
2. 状态与日志可机器读取（JSON）。
3. Agent 只需调用单一服务入口，不必直接处理多方 CalDAV 细节。

## 7. 当前边界

1. 不区分 work/personal 展示层标签（统一在 iCloud 目标日历观察）。
2. 不提供钉钉事件写回（符合“工作仅在钉钉改动”的约束）。
3. 状态存储当前为 JSON 文件（后续可升级 SQLite）。

## 8. 后续可扩展方向

1. 增加系统级守护部署模板（systemd/pm2）。
2. 引入 SQLite 与迁移脚本，提升状态管理能力。
3. 增加 webhook/通知（同步失败告警）。
4. 对外提供更细颗粒度 API，便于 Agent 精准操作。
