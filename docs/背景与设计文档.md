# 背景与设计文档

## 1. 项目背景

`calendar-sync` 用于把钉钉工作日程稳定同步到 iCloud，形成统一可读视图，降低人工维护和多系统查询成本。

核心业务方向是单向同步：

`钉钉 CalDAV -> iCloud CalDAV`

## 2. 设计目标

1. 生产可用：长期运行、可观测、可排障。
2. 配置可控：Web/CLI 都可操作，敏感信息脱敏展示。
3. 成本可控：日志支持“先简后详”读取，便于 Agent 节省 token。
4. 风险可控：删除有保护策略，避免误删放大。

## 3. 核心设计决策

### 3.1 日历选择模型

1. 钉钉来源日历：多选（`DINGTALK_CALDAV_SOURCE_CALENDAR_NAMES/HREFS`）。
2. iCloud 目标日历：单选（`ICLOUD_TARGET_CALENDAR_NAME/HREF`）。
3. 目标日历为空时：自动选择第一个可写日历（优先避开只读倾向日历）。

### 3.2 配置加载模型

1. 服务运行时只读取当前目录 `.env`。
2. 不再从 `test/` 或其他测试路径回退加载环境变量。

### 3.3 日志与排障模型

1. 日志基础分级：`error/warn/info/debug`。
2. 查询模式：
   - `summary`：简要（时间、级别、内容、ID）。
   - `detail`：详细（含 context）。
3. 定向检索：支持按 `level`、`id/ids`、`from/to`、`scope(startup/today/all)` 查询。

### 3.4 同步安全模型

1. 幂等更新：基于 UID 与 ETag 判断新建/更新/跳过。
2. 删除保护：
   - 连续 missing 轮次确认（`SYNC_DELETE_CONFIRM_RUNS`）。
   - 删除比例保护（`SYNC_DELETE_MAX_RATIO`）。
3. 演练模式：`SYNC_DRY_RUN=true` 时只计算不写入。

## 4. 系统结构

1. `src/server.js`：HTTP 服务、UI、定时调度、心跳。
2. `src/sync/sync-once.js`：同步主引擎（来源多选、目标单选）。
3. `src/cli.js`：运维命令入口（状态、日志、配置、同步）。
4. `src/lib/logger.js`：结构化日志读写。
5. `src/ui/*`：Web 控制台（状态、日志、配置编辑）。

## 5. 发布边界

1. 根目录建议仅保留两个主工程目录：`service/`、`skills/`。
2. 发布包不包含测试目录及临时目录（如 `test/`、`tesk/`）。
3. 发布前以部署文档为准执行连通性与运行检查。
